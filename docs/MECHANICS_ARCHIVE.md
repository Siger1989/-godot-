# MECHANICS ARCHIVE

## Purpose

`scenes/mvp/FourRoomMVP.tscn` is the current verification room for the backrooms demo. Treat it as a reusable mechanism sandbox, not disposable whitebox content.

Every accepted mechanic tested in this room should be recorded here so later rooms can reuse the same modules, scripts, materials, tuning values, and validation scripts without recovering decisions from chat history.

## Current Verification Room Roles

- `Room_A_Start`: player spawn, baseline movement, camera, animation, floor/wall scale.
- `Room_B_Light`: ceiling light, material readability, actor shadows, light flicker readability.
- `Room_C_Objective`: portal traversal, visibility/occlusion continuation, future objective placeholder.
- `Room_D_Narrow`: narrow pressure room, monster spawn, monster flee routing, door/wall snagging, floor edge checks.

## Reuse Rule

- Keep mechanisms modular and scene-owned. Prefer module scenes, components, data, and validation scripts over one large scene script.
- Runtime generation in `scripts/core/SceneBuilder.gd` and baked editor data in `scenes/mvp/FourRoomMVP.tscn` must stay functionally aligned.
- New maps should add areas, portals, markers, modules, and data. They should not fork the core behavior for a specific named room.
- Player, monster, lighting, foreground occlusion, and future visibility logic remain separate systems.
- Scene construction pieces are generated by component type. Future maps should place and tune instances through specs/data, not by forking mesh/collision/material generation for a room, direction, or one-off doorway.

## Implemented And Reusable Mechanisms

### Scene Layout And Geometry

- Main scene: `scenes/mvp/FourRoomMVP.tscn`.
- Runtime builder: `scripts/core/SceneBuilder.gd`.
- Layout data: `data/four_room_mvp_layout.yaml`.
- Room/portal/marker components live under `scripts/scene/`.
- Four rooms form the current closed loop: A -> B -> C -> D -> A.
- The saved and runtime scene shell is split by responsibility: `LevelRoot/Geometry` contains constructed floor/wall/ceiling/door-frame/light-panel geometry, while `LevelRoot/Areas` contains only the four room metadata nodes.
- `LevelRoot/Rooms` is a removed legacy mixed container. `SceneBuilder.gd` deletes it during build if it is found, and `ValidateCleanRebuildScene.gd` fails if it returns.
- Walls use explicit filler joints at important corners and T-junctions so visible seams do not rely on exact edge contact.
- Each portal opening is a selectable `WallOpening_P_*` body with one canonical local U-shaped visual mesh plus simple primitive collisions for left, right, and top. X/z orientation is handled by rotating the node, not by generating a different mesh path.
- Door frames are single canonical local U-shaped visual meshes generated by `scripts/scene/DoorFrameVisual.gd`. X/z orientation is handled by rotating the node, and door-frame nodes must keep `scale = Vector3.ONE`.
- Door frames are visual trim only; they do not add player-blocking collision.

Reuse notes:
- Keep simple primitive collision for wall openings and floor on mobile.
- Keep door-frame visual and wall-opening collision separate.
- If a future map changes room count, reuse portal and area data instead of adding per-room behavior branches.
- Keep room metadata and constructed geometry separated for future maps. Add new spatial metadata under `Areas`; add visible/colliding pieces under `Geometry`.
- Add new construction object types through one reusable generator/module path. Do not fix visual mismatches by scaling one instance or adding direction-specific mesh branches.

### Floor And Collision Stability

- Floor visuals are separate from floor physics.
- Floor visuals are regular per-room panels: `Floor_Room_A`, `Floor_Room_B`, `Floor_Room_C`, and `Floor_Room_D`.
- Per-room floor visuals are `MeshInstance3D` nodes in the `floor_visual` group. They own no collision and cast no shadows.
- Floor visual UVs are based on world coordinates so square floor texture scale stays consistent across room boundaries.
- One continuous `Floor_WalkableCollision` owns stable walkable physics across the four-room footprint.
- This avoids physics seams where the monster or player can drop at room boundaries while avoiding the old irregular south/north strip visual join.

Validation:
- `scripts/tools/ValidateCleanRebuildScene.gd`
- `scripts/tools/ValidateFloorCoverage.gd`
- Expected latest result: `FLOOR_COVERAGE_VALIDATION PASS samples=118 monster_y=0.063`

### Materials And Backrooms Look

- Wall material: `materials/backrooms_wall.tres`.
- Floor material: `materials/backrooms_floor.tres`.
- Door-frame material: `materials/backrooms_door_frame.tres`.
- Ceiling material: `materials/backrooms_ceiling.tres`.
- Ceiling light panel material: `materials/backrooms_ceiling_light.tres`.
- Texture assets live under `materials/textures/`.
- Generated U meshes include UVs so script-built geometry renders the same materials as baked geometry.
- Generated wall, ceiling, wall-opening, door-frame, and floor visual ArrayMeshes must follow `scripts/scene/GeneratedMeshRules.gd`: vertex, normal, UV, and tangent arrays are required when using the normal-mapped Backrooms materials.
- Ordinary walls and `WallJoint_*` filler blocks use `GeneratedMeshRules.build_box_mesh()` instead of `BoxMesh`, so they share the same render data rule as portal wall openings. Simple `BoxShape3D` collision is still kept separate for mobile stability.
- Portal wall-opening UVs use the same wall world-size texture rule as ordinary walls instead of a per-opening normalized UV rule.
- Generated visual meshes also set `material_override` to the expected material. This keeps ordinary walls/floors and generated U meshes on one render rule for editor preview, runtime rebuilds, and foreground-occlusion material restore.
- Generated vertical wall/opening/door-frame faces use one shared tangent basis in `GeneratedMeshRules.gd` (`Vector3.DOWN.cross(normal)`, positive tangent sign) so normal-mapped wall faces do not change brightness just because the wall direction changes.
- Generated vertical wall/opening/door-frame faces use upright UVs: higher local `y` maps to higher UV `v`. This keeps directional wallpaper/stain detail from rendering upside down across all walls.
- Per-room floor visuals use corrected top-face triangle order plus world-coordinate UVs, so the floor remains visible under the current backface-culling material rule and the tile pattern stays aligned.
- Wall, floor, door-frame, and ceiling materials use Lambert Wrap diffuse lighting for the Mobile renderer so the same wall rule does not produce isolated dark faces.
- Current normal-map strength is intentionally restrained for Mobile readability: wall `0.22`, floor `0.28`, and door frame `0.24`.
- Wall, floor, door-frame, ceiling, and foreground cutout materials use backface culling. Two-sided rendering is avoided for these room surfaces because it can make otherwise matching wall faces preview with inconsistent backside lighting.
- Foreground occlusion cutout materials use the same Lambert Wrap diffuse rule as the regular Backrooms materials, so an occluding wall does not become a different-looking wall while the local player-area cutout is active.
- A single low-strength `WorldEnvironment` is generated under `LevelRoot/Lights` to give all rooms a shared warm ambient baseline. Current ambient tuning is color `Color(1.0, 0.9, 0.66)`, energy `0.18`, and sky contribution `0.0`.
- The previous global wall-floor, wall-ceiling, and door-frame seam grime mesh pass was removed by user request on 2026-05-02 because it made the prototype visually cluttered.
- Do not add skirting/baseboard strips, wall-base dirt bands, ceiling-edge dirt bands, or door-frame seam dirt globally unless a later art pass is explicitly approved.
- `scripts/tools/ValidateSeamGrime.gd` now validates the removal state and expects no active seam-grime nodes or assets.

Reuse notes:
- Wall, wall opening, and door-frame materials should stay aligned unless a later art pass intentionally separates trim style.
- Texture scale/readability should be tuned in the material resources first before changing mesh dimensions.
- Keep scene-wide readability fixes in lighting/material rules. Do not hand-brighten one wall mesh or add transparent floor/wall overlays to fix a single dark face.
- If contact dirt is revisited later, prototype it locally in one test area and review it visually before applying it to every wall, ceiling, and doorway.

Validation:
- `scripts/tools/ValidateGeneratedMeshRules.gd`
- `scripts/tools/ValidateMaterialLightingRules.gd`
- `scripts/tools/ValidateSeamGrime.gd`
- Expected latest render-rule result: `GENERATED_MESH_RULES_VALIDATION PASS`
- Expected latest UV rule: vertical generated triangles increase UV `v` with height.
- Expected latest wall tangent diagnostic: vertical wall/opening tangent signs are unified to `+`
- Expected latest material-lighting result: `MATERIAL_LIGHTING_RULES_VALIDATION PASS`

### Ceiling Lights, Flicker, And Shadows

- Each room has a separate `Ceiling_Room_*` slab.
- Each room has one visible `CeilingLightPanel_Room_*` mesh and one real `OmniLight3D` under `LevelRoot/Lights`.
- Light panels and real lights are separate by design.
- Real room lights cast shadows.
- Static room visuals now use one type-level layer rule instead of room-specific visual layers. Floors, ceilings, solid walls, wall openings, door frames, and ceiling light panels all use `STATIC_GEOMETRY_LAYER`.
- All four room `OmniLight3D` nodes use `STATIC_GEOMETRY_LAYER | ACTOR_LIGHT_LAYER` for both `light_cull_mask` and `shadow_caster_mask`. Room names remain useful for placement and tuning, not for changing how one wall renders.
- Player and monster imported meshes are configured to cast shadows at runtime and use the actor light layer.
- Flicker is controlled by `scripts/lighting/LightingController.gd`.
- Flicker changes real `OmniLight3D.light_energy` and matching panel emission energy, then restores base values.
- Flicker is intentionally rare and random, not a fixed obvious interval.

Current tuning:
- Base ceiling light energy: `1.05`.
- Ceiling light range: `6.0`.
- Ceiling light attenuation: `0.78`.
- Ceiling light shadow bias: `0.02`.
- Ceiling light shadow normal bias: `0.35`.
- Ceiling light shadow opacity: `1.0`.
- Base ceiling light panel emission: `1.10`.
- Ambient fill: `WorldEnvironment` color `Color(1.0, 0.9, 0.66)`, energy `0.18`, sky contribution `0.0`.
- Static light mask: `257` (`STATIC_GEOMETRY_LAYER | ACTOR_LIGHT_LAYER`) on all four ceiling lights.
- Flicker startup delay: `18` to `45` seconds.
- Flicker cooldown: `28` to `70` seconds.
- Flicker chance after cooldown: `0.018` per second.
- Bright flicker spike range: `1.25` to `1.85`.

Validation:
- `scripts/tools/ValidateLightFlicker.gd`
- `scripts/tools/ValidateSceneShadows.gd`
- Expected latest shadow/coverage result: `SCENE_SHADOW_VALIDATION PASS baked=true runtime=true`

### Foreground Occlusion

- Controller: `scripts/camera/ForegroundOcclusion.gd`.
- Shader: `materials/foreground_occlusion_cutout.gdshader`.
- The system raycasts from camera to player.
- Hit foreground wall/opening/frame meshes stay visible but receive a local player-area cutout material.
- The cutout has a feathered edge so hidden and visible regions transition smoothly.
- Occlusion detection uses several camera-aligned target probes around the player, not just the center point.
- Detection is bidirectional to reduce misses when the camera is very close to or just past a wall plane.
- Cutouts use a short release delay (`cutout_release_delay = 0.16`) before restoring the original material, preventing one-frame wall flashes during forward/back movement across wall boundaries.
- Collisions stay active; this is visual-only and does not affect movement.
- Door frames are included visually so they do not float in front of the player after the wall opening receives a cutout.

Validation:
- `scripts/tools/ValidatePhase3Occlusion.gd`

### Four-Room Construction Visual Rule

- Builder: `scripts/core/SceneBuilder.gd`.
- Box mesh rule: `scripts/scene/GeneratedMeshRules.gd`.
- Portal wall body: `scripts/scene/WallOpeningBody.gd`.
- Door frame visual: `scripts/scene/DoorFrameVisual.gd`.
- Four rooms remain a uniform 2x2 grid of 6m x 6m spaces.
- Wall and portal-opening spans meet wall-joint blocks at their edges (`5.64m` span) instead of overlapping into the joint blocks.
- Solid wall and wall-joint visuals render side faces only; full box collision stays in place for movement stability.
- Portal wall bodies skip rendered floor/ceiling profile cap edges.
- Door-frame visuals skip rendered floor cap edges at the frame feet.
- Wall/joint and portal wall visuals extend slightly beyond floor/ceiling contact to hide black edge cracks without reintroducing coplanar horizontal faces.
- Portal wall bodies and door-frame visuals use explicit side-face UVs so narrow reveal/trim faces do not collapse to vertical texture stripes.
- This is intended to avoid z-fighting artifacts such as comb-like floor seams, vertical corner striping, black contact lines, stretched door side faces, and flashing contact faces.

Validation:
- `scripts/tools/ValidateGeneratedMeshRules.gd`
- `scripts/tools/ValidateCleanRebuildScene.gd`

### Monster MVP

- Monster scene: `scenes/modules/MonsterModule.tscn`.
- Monster model: `3D模型/guai1.glb`.
- Controller: `scripts/monster/MonsterController.gd`.
- Current states:
  - `WANDER`
  - `IDLE_LOOK`
  - `FLEE`
- Vision uses a forward horizontal cone plus physics line-of-sight raycast.
- Close panic detection can trigger flee even outside the forward cone.
- Flee uses room and portal nodes as lightweight waypoints before full navmesh work.
- Flee route scoring prefers exits farther from the player and avoids running into the player's current area when another route is available.
- If the monster hits a wall or makes too little progress, it repaths.
- Monster instance scale saved in the editor is preserved at runtime by moving only its global position.
- Current saved monster scale in the MVP scene: `(0.953989, 0.387199, 0.688722)`.
- Monster movement collision is intentionally smaller than the full visual silhouette to reduce door/wall snagging.
- Floor snap and last-safe-floor recovery reduce falling through edge cases.
- Monster Idle, Walk, and Run clips loop with root-motion position tracks disabled.
- Walk/Run playback direction follows local movement direction: if the body is moving backward relative to facing direction, leg animation plays in reverse.

Validation:
- `scripts/tools/InspectMonsterModel.gd`
- `scripts/tools/ValidateMonsterAI.gd`
- `scripts/tools/ValidateMonsterCollisionLimit.gd`
- `scripts/tools/ValidateMonsterSavedScale.gd`

Known limitation:
- This is still an MVP behavior layer. It has no full navmesh pathfinding, no attack, no damage, no door state logic, and no multiplayer synchronization.

## Archive Update Protocol

When a mechanism is accepted in this verification room:

1. Update this file with purpose, files, behavior, tuning knobs, validation scripts, known limits, and reuse notes.
2. Update `docs/PROGRESS.md` with the latest implementation and validation result.
3. Add a stable rule to `docs/DECISIONS.md` if the behavior should not be casually changed later.
4. Update root `CURRENT_STATE.md`.
5. Mirror important docs under `四房间MVP_Agent抗遗忘执行包/docs/`.
6. Put long command output in `logs/`.
